# 유스케이스: ZIASTACK DR 복구 자동화

## 액터

- **운영자**: ZIASTACK UI를 통해 복구를 요청하고 정책을 승인하는 엔지니어
- **AI 에이전트**: Claude API 기반. 정책 생성, 실행 제어, 리포트 작성 담당
- **OpenStack**: Nova/Neutron/Cinder/Glance API를 제공하는 인프라

---

## UC-01: 기존 타겟 서버로 복구 (정상 흐름)

**전제조건**
- 소스 서버가 장애 상태이고 복구 타겟 서버가 이미 존재한다
- 백업 이미지가 Glance에 존재한다

**흐름**
1. 운영자가 UI에서 소스 서버를 선택하고 복구 요청을 제출한다
2. AI 에이전트가 OpenStack에서 소스 서버의 리소스 현황을 수집한다
3. 시스템이 타겟 서버의 존재를 확인한다 → 존재함
4. AI 에이전트가 Chroma에서 관련 런북을 RAG 검색한다
5. AI 에이전트가 복구 정책을 생성한다 (백업 시점, 복구 유형, 실행 순서)
6. 정책이 UI에 표시되고 운영자가 승인한다
7. MCP → FastAPI → OpenStack SDK로 복구가 실행된다
8. 복구 진행 상태가 WebSocket으로 UI에 실시간 표시된다
9. 복구 완료 시 AI가 리포트를 생성하고 Slack/Jira로 발송한다
10. 복구 이력과 정책이 DB 및 Chroma에 저장된다

**예외**
- 4단계: RAG 검색 결과 없음 → AI가 수집 데이터만으로 정책 생성 후 경고 표시
- 7단계: OpenStack API 호출 실패 → 에러 메시지와 함께 blocked 상태로 중단, 운영자 알림

## UC-02: 신규 VM 생성 후 복구

**전제조건**
- 운영자가 입력한 서버 이름으로 OpenStack에서 VM 조회 시 존재하지 않는다
- 백업 이미지가 Glance에 존재한다

**흐름**
1. 운영자가 UI에서 소스 서버 이름을 입력하고 복구 요청을 제출한다
2. AI 에이전트가 OpenStack Nova API에서 해당 이름의 VM을 조회한다 → 존재하지 않음
3. AI 에이전트가 소스 서버 스펙(CPU/Memory/Disk/네트워크)을 수집한다
4. AI 에이전트가 소스 서버 스펙과 Chroma 런북을 기반으로 VM 파라미터 및 복구 정책을 함께 생성한다
5. 정책에 "신규 VM 생성 포함" 항목이 표시되고 운영자가 승인한다
6. MCP Server → FastAPI → Nova API로 VM이 자동 생성된다
7. Userdata 스크립트로 ZConverter Cloud AI 에이전트가 VM 내부에 자동 설치된다
8. VM 내 ZConverter AI 에이전트가 복구 작업을 자동으로 진행한다 (세부 동작 TBD)
9. 이후 UC-01의 8~10단계와 동일하게 진행된다

**예외**
- 6단계: VM 생성 실패 (리소스 부족) → 에러와 필요 리소스 정보를 운영자에게 전달 후 중단
- 7단계: Userdata 실행 실패 → 실패 로그 수집, 운영자에게 알림 후 blocked 상태로 전환 (TBD)

## UC-03: 운영자 정책 거절 → 재생성

**전제조건**
- UC-01 또는 UC-02의 5단계까지 완료, 정책이 UI에 표시된 상태

**흐름**
1. 운영자가 AI 생성 정책을 검토하고 거절한다
2. (선택) 운영자가 거절 사유를 입력한다
3. AI 에이전트가 거절 사유를 반영하여 정책을 재생성한다
4. 재생성된 정책이 UI에 다시 표시된다
5. 운영자가 승인하면 UC-01/02의 7단계로 이어진다

**예외**
- 3회 거절 시 작업이 blocked 상태로 중단되고 운영자에게 알림이 전송된다

---

## UC-04: 복구 실패 → AI 분석 → 재시도

**전제조건**
- 복구 실행이 시작되었으나 OpenStack 작업 중 오류가 발생했다

**흐름**
1. 복구 실행 중 오류 발생
2. AI 에이전트가 실패 로그를 자동 수집한다
3. AI 에이전트가 원인을 분석하고 해결 방안을 생성한다
4. 수정된 파라미터로 자동 재시도한다 (최대 3회)
5. 재시도 성공 시 UC-01의 9단계로 이어진다
6. 3회 모두 실패 시 AI가 분석 결과와 해결 방안을 운영자에게 제시하고 blocked 상태로 전환된다
7. 운영자가 수동 조치 후 재시도 가능

**예외**
- 재시도 중 발생한 새로운 오류도 각각 로그로 수집된다
- 최종 실패 지식이 Chroma에 누적된다

---

## UC-05: 복구 진행 중 운영자 강제 중단

**전제조건**
- 복구 실행이 진행 중인 상태

**흐름**
1. 운영자가 UI에서 복구 작업 중단을 요청한다
2. 시스템이 진행 중인 OpenStack 작업의 안전한 중단 가능 여부를 확인한다
3. 중단 가능한 경우 작업을 취소하고 cancelled 상태로 전환한다
4. 중단 불가능한 경우 (예: VM 생성 완료 직전) 현재 단계 완료 후 다음 단계 진입 전에 중단한다
5. 부분 완료된 리소스 목록이 운영자에게 표시된다
6. 이력이 DB에 저장된다

---

## 상태 전이 요약

```
pending
  → planning    : 운영자 복구 요청 접수
  → confirming  : AI 정책 생성 완료
  → executing   : 운영자 승인
  → completed   : 복구 성공
  → failed      : 복구 실패 (3회 재시도 후)
  → blocked     : 운영자 개입 필요 (Agent 설치 실패, 정책 3회 거절 등)
  → cancelled   : 운영자 강제 중단
```
