# Architecture Decision Records: ZIASTACK DR 복구 자동화

## 철학
복구 자동화의 신뢰성 최우선. AI가 틀릴 수 있다는 전제 하에, 실행 전 반드시 엔지니어 승인을 거친다.
외부 의존성은 명확한 인터페이스로 격리한다. MCP Server가 AI와 인프라 사이의 유일한 경계다.

---

### ADR-001: LLM — Claude API 선택 (로컬 LLM 미사용)

**결정**: 로컬 LLM(Ollama 등) 대신 Anthropic Claude API를 사용한다.

**이유**: 복구 정책 생성과 로그 분석은 높은 추론 품질이 필요하다. 로컬 LLM은 하드웨어 요구사항과 운영 부담이 크고, MVP 단계에서 모델 품질 리스크를 감수할 이유가 없다.

**트레이드오프**: 외부 API 의존성. 네트워크 장애 시 복구 자동화 불가. 추후 온프레미스 요구사항 발생 시 로컬 LLM으로 교체를 고려한다.

---

### ADR-002: MCP Server + FastAPI 2-레이어 분리

**결정**: AI Agent → MCP Server → FastAPI → OpenStack 구조로 인프라 접근을 2단계로 분리한다.

**이유**: MCP Server는 AI가 호출할 Tool 인터페이스를 정의하고, FastAPI는 실제 인증·상태관리·SDK 호출을 담당한다. 역할이 다르므로 분리해야 각각 독립적으로 테스트·교체할 수 있다.

**트레이드오프**: 레이어가 늘어나 지연(latency)이 증가한다. 단순 VM 조회도 2홉을 거쳐야 한다.

---

### ADR-003: 복구 타겟 서버 식별 — 서버 이름 기반

**결정**: 운영자가 입력한 서버 이름으로 Nova API를 조회해 타겟 VM 존재 여부를 판단한다. 해당 이름의 VM이 없으면 신규 생성 플로우로 분기한다.

**이유**: OpenStack 내 VM 식별자로 이름이 가장 직관적이고 운영자가 이미 인지하고 있는 정보다. UUID 기반은 운영자에게 불편하다.

**트레이드오프**: 동일 이름의 VM이 여러 개 존재할 경우 모호성 발생. 이 경우 시스템은 에러를 반환하고 운영자에게 명확한 식별자를 요청한다.

---

### ADR-004: Kafka — Execute Phase 이벤트 전용

**결정**: Kafka는 복구 실행(Execute Phase) 이벤트만 처리한다. Plan/Confirm 단계 이벤트는 Kafka를 거치지 않는다.

**이유**: Plan/Confirm 단계는 단발성 요청-응답 구조라 이벤트 큐가 불필요하다. Kafka 복잡도를 Execute Phase의 실시간 모니터링 목적으로만 한정해 운영 부담을 줄인다.

**트레이드오프**: Plan/Confirm 단계 이벤트 추적이 DB 로그에만 의존한다. 실시간 스트리밍이 필요해지면 재검토 필요.

---

### ADR-005: Confirm — 거절 사유 필수 입력

**결정**: 엔지니어가 AI 생성 정책을 거절할 때 사유 입력을 필수로 한다.

**이유**: 사유 없이 거절하면 AI가 사실상 동일한 정책을 재생성한다. 거절-재생성 루프가 3회를 낭비하고 결국 blocked가 되는 결과를 막기 위해 사유를 강제한다.

**트레이드오프**: 운영자에게 입력 부담을 준다. 긴급 상황에서 마찰이 될 수 있다. 추후 사유 선택지(체크박스) 추가를 고려한다.

---

### ADR-006: ZConverter AI 에이전트 설치 — Userdata 방식

**결정**: VM 생성 시 Nova Userdata 스크립트로 ZConverter Cloud AI 에이전트를 자동 설치한다.

**이유**: SSH 수동 설치는 자동화를 깨고 운영자 개입이 필요하다. Userdata는 VM 부팅 시 자동 실행되므로 완전 자동화가 가능하다.

**트레이드오프**: Userdata 실행 실패 시 감지가 어렵다. VM 상태는 ACTIVE이지만 Agent 미설치인 상태가 발생할 수 있다. 감지 방법은 TBD.
